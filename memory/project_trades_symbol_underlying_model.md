TODO #35 (Trades-table overhaul) — agreed underlying-symbol data model (design 2026-06-14):

- **`Symbol`** (rename of `Ssjacent`): normalized product **root**, equal to portfolio `Uxxx.symbol` for every instrument type — ticker for stocks (CA, 1976), root for futures/FOPs (CHF, CL, SOFR3, GC, ES, MSF, MCL), underlying for equity options (AAPL). The rename to a shared name was blocked until normalized, because today `Ssjacent` and `Uxxx.symbol` diverge for FX-futures/FOPs (Ssjacent `6SM6` vs portfolio `CHF`).
- **`Underlying`** (NEW column): raw IBKR **localSymbol** of the underlying futures contract (`6SM6`, `CLX5`, `SR3Z6`) — preserves the contract month losslessly. NULL for stocks/equity options. Named `Underlying`, not `Contract` (Contract is ambiguous vs the option contract).
- **Why the new column is required:** today the underlying future month lives ONLY inside `Ssjacent`'s localSymbol. `Instrument` and `Exp.Date` hold the OPTION expiry, not the future. Proof: `CLX5` (Nov-25 CL future) carries `Instrument="CL 16OCT25 60 P"` / `Exp.Date=16.10.2025` — an Oct option on the Nov future; normalizing `Ssjacent→CL` would lose "Nov". (6S/SOFR option & future months coincide, hiding the issue.) These divergent rows are options-on-futures (FOPs), not bare futures.
- **Normalization resolver (corrected 2026-06-14):** `Tickers` is NOT usable (holds only current contracts — CLX5/ESZ3/GCM5/etc. absent; TradingClass linkage inconsistent: SR3 works, CHF/CL don't). **BACKFILL uses `firstToken(Instrument)`** — the leading token of Instrument already equals the root for every FOP/future (CHF, CL, SOFR3, GC, ES, MSF) and the portfolio symbol. Detector (validated vs all 1210 distinct Ssjacent): normalize iff `Ssjacent` matches `^[A-Z0-9]{1,5}[FGHJKMNQUVXZ][0-9]{1,2}$` AND `Ssjacent != firstToken(Instrument)` (the `!=` guard excludes the lone false positive ESTX50 and legacy free-text rows). → `Symbol=firstToken(Instrument)`, `Underlying=Ssjacent`. **INGESTION resolver is OPEN**: `buildInstrumentName(sym,…)` (Tbasics/R/general.R:628) uses sym verbatim and the flex Instrument is built FROM Ssjacent (trade_operations.R:1099), so no root-bearing Instrument exists yet at import — need to know how 6S/CL/SOFR3 FOPs actually enter (flex UnderlyingSymbol value? manual modal?); a sample flex CSV row would settle it; likely a small static FX/futures map applied before buildInstrumentName.
- **JOIN SAFETY (verified 2026-06-14):** the canonical portfolio↔trade join in `Tdata/R/account.R` is **type-gated** — stocks join `symbol==Ssjacent`, everything else (FOPs/futures) joins on `Instrument` (unchanged by the rename), with `multiple="first"`. So normalizing `Ssjacent→Symbol` does NOT affect the FOP join there. BUT `Tuser/symbol/logic/symf.R::stats_one_position` is NOT type-gated: it left-joins BOTH `port_by_inst` (Instrument) AND `port_by_symbol` (`Ssjacent==symbol`) for all rows then `coalesce()`s. `port_by_symbol` is `distinct(symbol)` (no fan-out) and coalesce prefers the Instrument match, so held FOPs are fine — but a FOP whose specific leg is absent from the snapshot while its root IS held would, post-normalization, get an unrelated leg's mktValue injected via the symbol branch (today it correctly stays NA). **Phase-1 fix: type-gate symf.R's symbol-branch to stocks, mirroring account.R.**
- **How to apply:** at INGESTION (extend the existing `sub("SPXW","SPX", trade$Ssjacent)` normalization at `RReporting/app/R/trade_operations.R:1099`, the flex-import path where `Ssjacent = UnderlyingSymbol`) so future 6S trades land as `Symbol=CHF` + `Underlying=6SxN` automatically; + manual Open/Adjust modals get an Underlying field; + one-time backfill of the ~40 existing FOP rows (74 divergent rows total, but BBBY/TLRY/TECK/SCCO/RUT/etc. are just CLOSED positions = false positives, not normalization targets). `EUR.CHF`/`EUR.USD` (Strategy "Money", Instrument="CFD") are spot-forex CFDs → handled with the CASH/FX work (#35 part 2), separate mapping.
