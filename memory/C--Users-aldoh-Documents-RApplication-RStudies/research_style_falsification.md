When user proposes an alpha hypothesis (e.g. retail-call-flow lag, post-earnings RR25 drift), they want it tested honestly even if the result is null.

**Why:** During the IV-vs-spot-lag study (April 2026), user accepted multiple null/fragile results in succession (AAPL p=0.91, WMT-fragile-to-LB, pooled catalyst p=0.19, FOMC-effect-already-published) without pushing me to keep digging for a positive. Asked me to "look at academic papers" before celebrating the FOMC result — i.e., wanted reality-check before commitment, not validation.

**How to apply:**
- Always include a robustness/specification-search check before reporting a positive finding. The user explicitly values knowing if a result is fragile.
- Do not soften null results. Frame them as "thesis falsified" rather than "interesting hint, more research needed" — the user prefers the hard answer.
- When the user proposes a research extension, immediately ask "what would convince us this is wrong?" and design the test to potentially fail.
- Run a quick literature check (SSRN/Google Scholar) early in the work, not after pipeline-build, since the user expects to know what's already published.
- Trade economics back-of-envelope is the right closing gate: if the effect doesn't survive realistic bid-ask + commission, say so plainly.
